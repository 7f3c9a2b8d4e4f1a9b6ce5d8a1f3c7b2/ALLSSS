# Audit Report

## Title
Missing Consensus Behavior Validation Allows Malicious Miner to Force Premature Term Change

## Summary
A malicious miner can force premature term transitions by claiming `NextTerm` behavior when `NextRound` should be used. The on-chain validation only verifies that term numbers increment correctly but fails to re-verify the `NeedToChangeTerm` criteria that determines whether a term change is actually warranted, allowing invalid consensus state transitions.

## Finding Description

The vulnerability exists due to a separation between off-chain behavior determination and on-chain behavior validation:

**Off-chain Behavior Determination:**
The `MainChainConsensusBehaviourProvider` determines whether to use `NextTerm` or `NextRound` behavior by checking if at least 2/3 of miners have `ActualMiningTimes` indicating a term change is needed. [1](#0-0) 

The `NeedToChangeTerm()` method enforces this 2/3 threshold. [2](#0-1) 

**Missing On-chain Validation:**
During block validation, `ValidationForNextTerm()` only checks that the term number increments by 1, but does NOT re-verify `NeedToChangeTerm()`. [3](#0-2) 

**Exploitable Entry Point:**
The `GetConsensusBlockExtraData` method accepts the miner-provided `Behaviour` field without validation and switches on it to generate consensus data. [4](#0-3) 

**Attack Sequence:**
1. Malicious miner modifies their node software to set `Behaviour = NEXT_TERM` in the `AElfConsensusTriggerInformation` message
2. The behavior field is defined in the protobuf as a settable field [5](#0-4) 
3. The miner's modified client-side code (normally in `AEDPoSTriggerInformationProvider`) sets arbitrary behavior [6](#0-5) 
4. Validation passes because only term number increment is checked [7](#0-6) 
5. `ProcessNextTerm` executes, updating term number, triggering elections and rewards [8](#0-7) 

## Impact Explanation

**Consensus Integrity Violation:**
- Forces premature term transitions, violating AEDPoS consensus rules that require 2/3 miner agreement
- Disrupts the round/term state machine, causing desynchronization between expected and actual term numbers
- Triggers election processes prematurely, potentially manipulating miner selection timing

**Economic Impact:**
- Term changes trigger reward distribution and treasury release tied to term boundaries [9](#0-8) 
- Premature term changes manipulate when rewards are distributed, potentially allowing timing-based exploitation
- Election snapshots are taken at term boundaries, affecting vote-weight calculations [10](#0-9) 

**Operational Impact:**
- Honest nodes following correct consensus rules may diverge from the manipulated chain
- Cross-chain operations and term-based contract logic operate on incorrect term values
- Miner list updates occur prematurely [11](#0-10) 

## Likelihood Explanation

**High Feasibility:**
- Attacker only needs to be an elected miner (achievable through normal voting process)
- Attack requires simple modification of client-side node software to set arbitrary `Behaviour` value
- No cryptographic breaks, timing constraints, or race conditions required

**Low Complexity:**
- Single malicious block with wrong behavior triggers the state corruption
- The validation gap is deterministic and always exploitable
- No economic costs beyond normal miner operation

**Difficult Detection:**
- Validation appears to pass successfully from external perspective
- Term number increment is valid, only the timing is premature
- Requires deep knowledge of consensus rules to detect the manipulation

## Recommendation

Add validation in `ValidationForNextTerm()` to re-verify that `NeedToChangeTerm()` returns true:

```csharp
private ValidationResult ValidationForNextTerm(ConsensusValidationContext validationContext)
{
    var extraData = validationContext.ExtraData;
    var validationResult = ValidationForNextRound(validationContext);
    if (!validationResult.Success) return validationResult;

    // Is next term number correct?
    if (validationContext.BaseRound.TermNumber.Add(1) != extraData.Round.TermNumber)
        return new ValidationResult { Message = "Incorrect term number for next round." };
    
    // NEW: Verify that term change is actually needed
    var blockchainStartTimestamp = // Get from state
    var periodSeconds = // Get from state
    if (!validationContext.BaseRound.NeedToChangeTerm(blockchainStartTimestamp, 
        validationContext.BaseRound.TermNumber, periodSeconds))
        return new ValidationResult { Message = "Term change not warranted by NeedToChangeTerm criteria." };
    
    return new ValidationResult { Success = true };
}
```

This ensures on-chain validation re-checks the same criteria used for off-chain behavior determination.

## Proof of Concept

A test demonstrating this vulnerability would:
1. Set up a consensus state where `NeedToChangeTerm()` returns false
2. Construct a block with `NextTerm` behavior despite this
3. Show that validation passes
4. Verify that `ProcessNextTerm` executes and updates term number prematurely

The vulnerability is confirmed by examining the validation code path which shows no call to `NeedToChangeTerm()` during `ValidationForNextTerm()`.

## Notes

The vulnerability stems from trusting client-side behavior determination without server-side re-validation. While honest nodes use `GetConsensusCommand` to determine correct behavior, a malicious miner can bypass this recommendation by modifying the `AElfConsensusTriggerInformation.behaviour` field in their node software. The on-chain contract must defensively validate all miner-controlled inputs, including the claimed behavior type.
