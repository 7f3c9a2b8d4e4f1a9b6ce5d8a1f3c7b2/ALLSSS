# Audit Report

## Title
Term Number Inconsistency in NextRound Validation Causes Permanent Consensus State Corruption and Term Transition Denial of Service

## Summary
The `ValidationForNextRound()` method fails to validate that the term number remains unchanged during NextRound transitions. This allows any miner producing a NextRound extra block to inject an arbitrary term number into consensus state, creating permanent inconsistency between `State.CurrentTermNumber.Value` and the Round object's term number, which causes all future legitimate term transitions to fail with transaction reverts.

## Finding Description

**Root Cause - Missing Term Number Validation:**

The `ValidationForNextRound()` method only validates round number increment and InValue nullness, completely omitting term number validation: [1](#0-0) 

In contrast, `ValidationForNextTerm()` explicitly validates term number increment: [2](#0-1) 

**Validation Pipeline Gap:**

For NextRound behavior, the validation pipeline adds only two validators, neither of which checks term number: [3](#0-2) 

The validation context includes `State.CurrentTermNumber.Value` but validators don't compare it against the round's term number: [4](#0-3) 

**State Update Asymmetry:**

`ProcessNextRound` converts input to Round via `ToRound()` which preserves the TermNumber field: [5](#0-4) 

However, `ProcessNextRound` updates only `State.CurrentRoundNumber` but never calls `TryToUpdateTermNumber`: [6](#0-5) 

In contrast, `ProcessNextTerm` properly calls both state update methods: [7](#0-6) 

**Post-Execution Validation Bypass:**

`ValidateConsensusAfterExecution` retrieves the current round and compares hashes, but since both the stored state and header contain the same malicious term number after execution, validation passes: [8](#0-7) 

The hash computation includes TermNumber in `GetCheckableRound()`: [9](#0-8) 

**Attack Vector:**

When generating NextRound consensus data, the term number is copied from current round: [10](#0-9) 

A malicious miner obtains this data and modifies the TermNumber field before broadcasting: [11](#0-10) 

## Impact Explanation

**Critical Severity - Permanent Consensus DoS:**

After the attack succeeds, the consensus state becomes permanently corrupted:
- `State.CurrentTermNumber.Value` = X (correct value)
- `State.Rounds[currentRound].TermNumber` = Y (attacker-controlled value)

When the next legitimate NextTerm transition occurs, it generates the first round of the new term by incrementing the current round's term number: [12](#0-11) 

This is invoked via: [13](#0-12) 

The NextTerm transaction then attempts to update `State.CurrentTermNumber.Value` using the corrupted term number + 1, which fails validation in `TryToUpdateTermNumber`: [14](#0-13) 

Since `ProcessNextTerm` has an Assert on this operation, the transaction reverts, **permanently preventing all future term transitions** - a complete consensus mechanism breakdown.

**Secondary Impacts:**
- Cross-contract synchronization failures between Consensus and Election contracts
- Reward distribution timing errors due to term boundary confusion
- Inability to update miner lists through normal term transitions

## Likelihood Explanation

**High Likelihood:**

**Attacker Requirements:**
- Must be a valid miner in the current round (standard mining permission)
- Must be scheduled to produce the NextRound extra block (deterministic from consensus)
- No special privileges beyond normal miner status required

**Attack Complexity: Low**
1. Wait for scheduled NextRound extra block production turn
2. Call `GetConsensusExtraData` to obtain legitimate NextRound consensus data
3. Parse the returned `NextRoundInput` and modify the `TermNumber` field to any value
4. Include modified data in block header's consensus extra data
5. Broadcast the block

**No Economic Barriers:**
- No additional cost beyond normal block production
- No slashing risk since all validations pass
- Attack is deterministic and always succeeds when executed

**Detection Difficulty:**
- Both before and after-execution validations pass
- Requires monitoring the specific divergence between `State.CurrentTermNumber.Value` and `State.Rounds[N].TermNumber`
- No events or logs indicate the inconsistency

## Recommendation

Add explicit term number validation in `ValidationForNextRound()`:

```csharp
private ValidationResult ValidationForNextRound(ConsensusValidationContext validationContext)
{
    var extraData = validationContext.ExtraData;
    
    // Validate round number increment
    if (validationContext.BaseRound.RoundNumber.Add(1) != extraData.Round.RoundNumber)
        return new ValidationResult { Message = "Incorrect round number for next round." };
    
    // ADD THIS: Validate term number remains unchanged
    if (validationContext.BaseRound.TermNumber != extraData.Round.TermNumber)
        return new ValidationResult { Message = "Term number must remain unchanged during NextRound transition." };
    
    // Validate InValues are null
    return extraData.Round.RealTimeMinersInformation.Values.Any(m => m.InValue != null)
        ? new ValidationResult { Message = "Incorrect next round information." }
        : new ValidationResult { Success = true };
}
```

Additionally, consider adding a sanity check in `ProcessNextRound` to assert that the input's term number matches `State.CurrentTermNumber.Value`.

## Proof of Concept

This vulnerability requires integration testing with the full AEDPoS consensus mechanism. A minimal POC would:

1. Deploy AEDPoS contract with multiple miners
2. Progress through several rounds until a miner is scheduled for NextRound
3. Intercept the legitimate NextRound consensus data generation
4. Modify the `TermNumber` field to `currentTermNumber + 1`
5. Submit the block with modified consensus data
6. Verify `State.CurrentTermNumber.Value` â‰  `State.Rounds[currentRound].TermNumber`
7. Attempt a legitimate NextTerm transition
8. Observe transaction revert in `TryToUpdateTermNumber` due to term number mismatch

The vulnerability is confirmed through code analysis showing the missing validation and asymmetric state update logic.

    public override ValidationResult ValidateConsensusAfterExecution(BytesValue input)
    {
        var headerInformation = new AElfConsensusHeaderInformation();
        headerInformation.MergeFrom(input.Value);
        if (TryToGetCurrentRoundInformation(out var currentRound))
        {
            if (headerInformation.Behaviour == AElfConsensusBehaviour.UpdateValue)
                headerInformation.Round =
                    currentRound.RecoverFromUpdateValue(headerInformation.Round,
                        headerInformation.SenderPubkey.ToHex());

            if (headerInformation.Behaviour == AElfConsensusBehaviour.TinyBlock)
                headerInformation.Round =
                    currentRound.RecoverFromTinyBlock(headerInformation.Round,
                        headerInformation.SenderPubkey.ToHex());

            var isContainPreviousInValue = !currentRound.IsMinerListJustChanged;
            if (headerInformation.Round.GetHash(isContainPreviousInValue) !=
                currentRound.GetHash(isContainPreviousInValue))
            {
                var headerMiners = headerInformation.Round.RealTimeMinersInformation.Keys;
                var stateMiners = currentRound.RealTimeMinersInformation.Keys;
                var replacedMiners = headerMiners.Except(stateMiners).ToList();
                if (!replacedMiners.Any())
                    return new ValidationResult
                    {
                        Success = false, Message =
                            "Current round information is different with consensus extra data.\n" +
                            $"New block header consensus information:\n{headerInformation.Round}" +
                            $"Stated block header consensus information:\n{currentRound}"

                var newMiners = stateMiners.Except(headerMiners).ToList();

                        State.ElectionContract.GetNewestPubkey.Call(new StringValue { Value = miner }).Value)
                    .ToList();

                    newMiners.Count == officialNewestMiners.Count &&
                    newMiners.Union(officialNewestMiners).Count() == newMiners.Count,
                    "Incorrect replacement information.");
            }
        }

        return new ValidationResult { Success = true };
    }

    internal Round GenerateFirstRoundOfNewTerm(int miningInterval,
        Timestamp currentBlockTime, long currentRoundNumber = 0, long currentTermNumber = 0)
    {
        var sortedMiners =
            (from obj in Pubkeys
                    .ToDictionary<ByteString, string, int>(miner => miner.ToHex(), miner => miner[0])
                orderby obj.Value descending
                select obj.Key).ToList();

        var round = new Round();

        for (var i = 0; i < sortedMiners.Count; i++)
        {
            var minerInRound = new MinerInRound();

            // The first miner will be the extra block producer of first round of each term.
            if (i == 0) minerInRound.IsExtraBlockProducer = true;

            minerInRound.Pubkey = sortedMiners[i];
            minerInRound.Order = i + 1;
            minerInRound.ExpectedMiningTime =
                currentBlockTime.AddMilliseconds(i.Mul(miningInterval).Add(miningInterval));
            // Should be careful during validation.
            minerInRound.PreviousInValue = Hash.Empty;

            round.RealTimeMinersInformation.Add(sortedMiners[i], minerInRound);
        }

        round.RoundNumber = currentRoundNumber.Add(1);
        round.TermNumber = currentTermNumber.Add(1);
        round.IsMinerListJustChanged = true;

        return round;
    }
