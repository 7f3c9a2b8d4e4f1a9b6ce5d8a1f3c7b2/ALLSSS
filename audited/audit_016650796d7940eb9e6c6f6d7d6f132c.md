### Title
Insufficient Connector Weight Validation Allows Numerically Unstable Bancor Calculations

### Summary
The TokenConverter contract's weight validation only enforces `0 < weight < 1` without a reasonable minimum threshold, allowing governance to initialize connectors with extremely small weights (e.g., 0.0001). Such weights cause the Bancor pricing formulas to fail due to numerical overflow/underflow in the exponential calculations, potentially resulting in tokens being priced at near-zero values or transaction failures.

### Finding Description

The connector weight validation in `TokenConverterContract.cs` uses `IsBetweenZeroAndOne()` which only checks that weights are strictly between 0 and 1, with no minimum threshold: [1](#0-0) 

This validation is applied in three critical functions:
- `Initialize()` for initial connector setup [2](#0-1) 

- `UpdateConnector()` for modifying existing connectors [3](#0-2) 

- `AddPairConnector()` for adding new connector pairs [4](#0-3) 

**Root Cause**: The Bancor pricing formulas use exponential calculations with weight ratios: [5](#0-4) [6](#0-5) 

The `Exp()` function implementation uses a Taylor series with only 20 iterations: [7](#0-6) 

When connector weights create extreme ratios (e.g., `0.5 / 0.0001 = 5000`), the exponential calculations become:
- For large ratios: `Exp(5000 * Ln(x))` where `Ln(x) â‰ˆ 0.1`, resulting in `Exp(500+)` - astronomically large numbers causing overflow or severe precision loss
- For small ratios: `Exp(0.0002 * Ln(x))` - results near 1.0, making tokens essentially free

### Impact Explanation

**Concrete Harm**:
1. **Token Mispricing**: Extremely small weight ratios cause the Bancor formula to return values approaching 1.0, making tokens purchasable at near-zero cost, allowing reserve drainage
2. **Calculation Overflow**: Large weight ratios cause exponential overflow, resulting in transaction failures and DoS of the swap functionality
3. **Precision Loss**: The 20-iteration Taylor series is insufficient for large exponents, producing mathematically incorrect prices even if overflow doesn't occur

**Affected Parties**: All users attempting to buy/sell tokens through misconfigured connectors, and the protocol treasury which holds reserves.

**Severity**: Medium - While requiring governance action to introduce the misconfiguration, once present, the impact allows systematic mispricing that could drain reserves or DoS trading operations.

### Likelihood Explanation

**Preconditions**:
- Connector Controller (Parliament by default) must approve connector initialization with extremely small weights
- This could occur through configuration error or malicious governance proposal

**Attack Complexity**: 
- The initial misconfiguration requires governance approval (not attacker-controlled)
- However, once misconfigured, any user can exploit mispricing through normal `Buy`/`Sell` operations [8](#0-7) [9](#0-8) 

**Feasibility**: Medium-Low likelihood due to governance requirement, but represents a missing safety guardrail against configuration errors. The production system uses 0.005 as minimum weight, but no code enforcement prevents smaller values. [10](#0-9) 

### Recommendation

**Code-Level Mitigation**:
1. Add minimum weight threshold validation (e.g., `weight >= 0.001m`) in `AssertValidConnectorWeight()`:

```csharp
private void AssertValidConnectorWeight(Connector connector)
{
    var weight = AssertedDecimal(connector.Weight);
    Assert(weight >= 0.001m && weight < decimal.One, 
        "Connector weight must be between 0.001 and 1.");
    connector.Weight = weight.ToString(CultureInfo.InvariantCulture);
}
```

2. Add similar checks in `IsBetweenZeroAndOne()` or create separate validation for weight-specific constraints

3. Consider adding maximum weight ratio validation between paired connectors to prevent extreme exponential calculations

**Test Cases**:
- Test connector initialization with weight = 0.0001 (should fail)
- Test Buy/Sell operations with weight ratios > 1000 to verify numerical stability
- Test boundary conditions at minimum threshold (0.001)

### Proof of Concept

**Initial State**:
- Connector A: Symbol="TOKEN_A", Weight="0.0001", Balance=1,000,000
- Connector B: Symbol="TOKEN_B", Weight="0.5", Balance=1,000,000

**Exploitation Steps**:
1. Parliament approves proposal to add connector pair with weights as above (via `AddPairConnector`)
2. User calls `Buy` to purchase 100,000 TOKEN_B tokens
3. Bancor calculation attempts: `y = 0.5 / 0.0001 = 5000`, then `Exp(5000 * Ln(x))`
4. Result: Either overflow/revert OR severe mispricing due to insufficient Taylor series precision

**Expected Result**: Transaction should fail validation due to minimum weight check

**Actual Result**: Validation passes, leading to calculation errors or exploitable mispricing

**Notes**

While this vulnerability requires governance action to introduce, it represents a missing safety control that could prevent catastrophic configuration errors. The lack of minimum weight bounds creates systemic risk even under honest governance due to potential configuration mistakes. This is classified as Medium severity following standard audit classifications where missing admin input validation can lead to user fund loss.

    public static long GetReturnFromPaid(long fromConnectorBalance, decimal fromConnectorWeight,
        long toConnectorBalance, decimal toConnectorWeight, long paidAmount)
    {
        if (fromConnectorBalance <= 0 || toConnectorBalance <= 0)
            throw new InvalidValueException("Connector balance needs to be a positive number.");

        if (paidAmount <= 0) throw new InvalidValueException("Amount needs to be a positive number.");

        decimal bf = fromConnectorBalance;
        var wf = fromConnectorWeight;
        decimal bt = toConnectorBalance;
        var wt = toConnectorWeight;
        decimal a = paidAmount;
        if (wf == wt)
            // if both weights are the same, the formula can be reduced
            return (long)(bt / (bf + a) * a);

        var x = bf / (bf + a);
        var y = wf / wt;
        return (long)(bt * (decimal.One - Exp(y * Ln(x))));
    }

    public static long GetAmountToPayFromReturn(long fromConnectorBalance, decimal fromConnectorWeight,
        long toConnectorBalance, decimal toConnectorWeight, long amountToReceive)
    {
        if (fromConnectorBalance <= 0 || toConnectorBalance <= 0)
            throw new InvalidValueException("Connector balance needs to be a positive number.");

        if (amountToReceive <= 0) throw new InvalidValueException("Amount needs to be a positive number.");

        decimal bf = fromConnectorBalance;
        var wf = fromConnectorWeight;
        decimal bt = toConnectorBalance;
        var wt = toConnectorWeight;
        decimal a = amountToReceive;
        if (wf == wt)
            try
            {
                // if both weights are the same, the formula can be reduced
                return (long)(bf / (bt - a) * a);
            }
            catch
            {
                throw new AssertionException("Insufficient account balance to deposit");
            }

        var x = bt / (bt - a);
        var y = wt / wf;
        return (long)(bf * (Exp(y * Ln(x)) - decimal.One));
    }

    {
        /*
        exp(y) = 1 + y + y^2/2 + x^3/3! + y^4/4! + y^5/5! + ...
        */

        decimal result = 1;
        while (iteration > 0)
        {
            //uint fatorial = Factorial(iteration);

            result += Pow(y, (uint)iteration) / fatorial;
            iteration--;
        }
